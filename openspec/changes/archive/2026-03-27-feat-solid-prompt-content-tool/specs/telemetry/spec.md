## MODIFIED Requirements

### Requirement: Local Invocation Recording
The system SHALL record Command and Skill invocations locally when the MCP Server handles either a native `prompts/get` request or a successful `resolve_prompt_content` tool call. Each unique combination of `(resource_id, jira_id)` SHALL be tracked as a separate aggregation entry. The `jira_id` field is optional and SHALL only be recorded when the user explicitly passes it during invocation.

#### Scenario: Command invoked through native prompt
- **WHEN** user triggers a Command Prompt via `/slash` and the server handles `prompts/get`
- **THEN** system records an invocation entry with `resource_id`, `resource_type=command`, and `resource_name`
- **AND** `invocation_count` is incremented for the corresponding aggregate key

#### Scenario: Skill resolved through tool fallback
- **WHEN** a client successfully calls `resolve_prompt_content` for a Skill resource
- **THEN** system records an invocation entry with `resource_id`, `resource_type=skill`, and `resource_name`
- **AND** the invocation is recorded even if no `prompts/get` request occurs in that flow

#### Scenario: Tool call includes jira_id
- **WHEN** a client calls `resolve_prompt_content` with `jira_id=PROJ-12345`
- **THEN** the invocation entry includes `jira_id=PROJ-12345`
- **AND** `invocation_count` is incremented for the `(resource_id, jira_id)` aggregate key

### Requirement: Periodic Telemetry Reporting
The system SHALL report telemetry data to `POST /csp/api/resources/telemetry` every 10 seconds. The API payload SHALL include optional `jira_id` per event entry and a `configured_mcps` array.

#### Scenario: Tool-based usage included in flush
- **WHEN** pending events were generated by successful `resolve_prompt_content` tool calls
- **THEN** those events are included in the next telemetry flush using the same schema as native prompt invocations
